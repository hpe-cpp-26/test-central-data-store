Cricket Auction System Documentation
This documentation outlines the architecture, data models, and logic for a Cricket Auction System. This system is designed to facilitate real-time player bidding, squad management, and financial constraint tracking (budget caps) for cricket leagues (e.g., IPL, BBL).
1. System Overview
The Cricket Auction System is a real-time distributed application that manages the registration of players, the bidding lifecycle, and team budget allocations.
Key Objectives
Real-time Bidding: Ensure sub-second latency for bid submissions and state synchronization across all franchise consoles.
Consistency: Guarantee that a player cannot be sold to multiple teams and that no team can breach its budget cap or squad size limits.
Transparency: Maintain an immutable audit log of all bids placed during the auction process.

3. The Auction State Machine
Every player brought to the hammer transitions through a strict lifecycle to prevent race conditions during concurrent bidding.
State Transitions
UNSOLD to BIDDING_IN_PROGRESS: Triggered when the auctioneer introduces the player. The current bid is initialized to the player's base_price.
BIDDING_IN_PROGRESS to BIDDING_IN_PROGRESS: Triggered whenever a valid higher bid is submitted within the countdown window. Each valid bid resets the countdown clock.
BIDDING_IN_PROGRESS to SOLD: Triggered when the countdown timer reaches zero after a bid has been made. The player is assigned to the highest bidder, and the franchise's remaining budget is decremented.
BIDDING_IN_PROGRESS to UNSOLD: Triggered if the countdown timer expires with zero bids placed. The player returns to the unsold pool for potential re-auction later.
4. Business Rules & Validation Logic
Before a bid is processed and broadcasted, it must clear the following validation checklist:
Incremental Progression: A new bid must exceed the current highest bid by a designated increment minimum (e.g., $+10\text{k}$ up to a specific tier, $+50\text{k}$ above it).
Budget Ceiling Violation:
$$\text{Bid Amount} \le \text{Franchise's Remaining Budget}$$
Squad Slots Check:
$$\text{Current Squad Count} < \text{Max Squad Size}$$
Overseas Cap Rule: If is_overseas == True, the franchise's overseas_count must be strictly less than the tournament's maximum overseas limit per team.
5. Architectural Considerations
Real-Time Synchronization
WebSockets / gRPC: Used to push real-time bid updates to all connected team dashboards simultaneously.
Distributed Cache (e.g., Redis): The active player's profile, current highest bid, and leading franchise ID should be held in-memory to handle heavy read/write throughput during active bidding.
Concurrency Control
To eliminate race conditions where two franchises click "bid" at the exact same millisecond:
Use Optimistic Concurrency Control (OCC) or Distributed Locking (via Redis Redlock) tied to the player_id.
Bids must process sequentially through a single-threaded queue per player instance to guarantee absolute ordering of events
