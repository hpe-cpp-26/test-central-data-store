Payment System Troubleshooting Guide

1.  Introduction This document outlines the standard operating procedures and guidelines for troubleshooting issues within our payment processing system. It is designed for Level 1/Level 2 support engineers, DevOps, and backend developers to diagnose and resolve payment failures efficiently.
    
2.  High-Level Architecture Understanding the flow of a payment is crucial for identifying where the failure occurred:
    
3.  Client/Checkout: User submits payment details.
    
4.  Payment Gateway: Securely routes the transaction (e.g., Stripe, PayPal, Adyen).
    
5.  Payment Processor: Handles the transaction logic and connects to card networks.
    
6.  Acquiring Bank: The merchant's bank which receives the funds.
    
7.  Issuing Bank: The customer's bank (approves or declines the transaction).
    
8.  Common Error Categories
    

A. Network & Connectivity Issues Symptoms: Timeouts, 502 Bad Gateway, 504 Gateway Timeout. Troubleshooting:

-   Check status pages of third-party payment gateways (e.g., [http://status.stripe.com](http://status.stripe.com) ).
    
-   Verify internal API connectivity and outbound firewall rules.
    
-   Inspect network latency metrics in monitoring tools (e.g., Datadog, New Relic).
    

B. Authentication & Authorization Symptoms: 401 Unauthorized, 403 Forbidden. Troubleshooting:

-   Verify API keys (ensure Test vs. Live mode are not mixed up).
    
-   Check if API keys have been recently rotated, revoked, or expired.
    
-   Ensure the merchant account is active and in good standing with the provider.
    

C. Data Validation Errors Symptoms: 400 Bad Request, Invalid Card Number, Missing Field. Troubleshooting:

-   Validate the payload sent to the payment gateway.
    
-   Check for malformed JSON, missing required fields (e.g., amount, currency), or invalid data types.
    
-   Review frontend client-side validation logic.
    

D. Fraud & Risk Rejections Symptoms: Card Declined, High Risk, Radar Blocked. Troubleshooting:

-   Check the fraud prevention dashboard (e.g., Stripe Radar, Kount, Signifyd).
    
-   Verify CVV (Card Verification Value) and AVS (Address Verification System) match status.
    
-   Review if the user's IP, email, or issuing country is on a blocklist or velocity rule trigger.
    

E. Processing & Bank Declines Symptoms: Insufficient Funds, Do Not Honor, Expired Card. Troubleshooting

-   These are usually soft or hard declines from the issuing bank.
    
-   Soft Declines: (e.g., Insufficient funds, temporary hold) - Prompt user to try again later or use another card.
    
-   Hard Declines: (e.g., Lost/Stolen card, account closed) - Block retry attempts immediately and notify the user to contact their bank.
    

4.  Standard Operating Procedure (SOP) for Incident Response Step 1: Identify the Scope - Is this affecting a single user, a specific payment method (e.g., Apple Pay failing), or is it system-wide? Step 2: Check Logs - Look for Trace IDs or Transaction IDs in Kibana, Splunk, or CloudWatch. Step 3: Correlate with Gateways - Cross-reference internal logs with the Payment Gateway's dashboard logs using the gateway's unique charge\_id or payment\_intent\_id. Step 4: Isolate the Component - Did it fail before reaching the gateway? At the gateway? Or during the webhook callback? Step 5: Remediate - Code issue? Roll back recent deployments or feature flags. Gateway issue? Route traffic to a backup gateway if your system supports multi-acquiring. Step 6: Post-Mortem - Document the root cause, time to resolution (TTR), and add missing alerts.
    
5.  Webhook Troubleshooting Webhooks are critical for asynchronous payment updates (e.g., payment\_intent.succeeded). Issue: Payment succeeded at gateway, but order status is still Pending in our system. Resolution:
    

-   Check if the gateway is attempting to send the webhook (look for 4xx/5xx responses from our webhook endpoint).
    
-   Verify webhook signatures are validating correctly in the backend.
    
-   Check for dead-letter queues (DLQ) if using async processing (Kafka/RabbitMQ).
    
-   Manually retry the webhook payload from the provider's dashboard.
    

6.  Escalation Matrix
    

-   Tier 1 (Customer Support): Handle basic user-facing declines (insufficient funds, expired cards, mismatched billing addresses).
    
-   Tier 2 (Platform Ops): Investigate API key issues, webhook delivery failures, and single-gateway minor outages.
    
-   Tier 3 (Engineering): Debug core backend logic failures, database transaction deadlocks, race conditions, and memory leaks.
    
-   External Escalation: Open a critical support ticket with the Payment Gateway support team (e.g., Enterprise support).
