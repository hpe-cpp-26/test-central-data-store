System Requirements Document: URL Shortener
This document specifies the functional and non-functional requirements for designing and implementing a production-grade, highly scalable URL shortener service (similar to Bitly or TinyURL).
1. Functional Requirements
Functional requirements define the specific behaviors, services, and operations that the system must provide to its end-users and administrators.
Core Redirection & Shortening APIs
URL Shortening: The system must accept a valid, long URL string (e.g., [https://www.example.com/deep/path/to/resource?ref=xyz](https://www.example.com/deep/path/to/resource?ref=xyz)) and return a unique, shortened alias URL (e.g., [https://pico.ly/abc12378](https://pico.ly/abc12378)).
Custom Aliases: Users should have the option to provide a custom, available string for their short link (e.g., [https://pico.ly/my-promo-campaign](https://pico.ly/my-promo-campaign)) instead of a system-generated token.
302 Redirect (HTTP): When a user navigates to a shortened URL, the system must immediately redirect the browser to the original long URL via an HTTP 302 Found (temporary redirect) or 301 Moved Permanently response status code.
Link Expiration: Users must be able to specify an optional expiration date and time for their shortened links. Once expired, navigating to the link should return a standard 410 Gone or 404 Not Found HTTP status page.
User Account & Asset Management
User Registration & Authentication: The system must support user sign-up, login, and secure session management via OAuth2 providers (Google, GitHub) or standard email/password authentication.
User Dashboard: Authenticated users must have a dashboard to view, edit, or manually delete their active shortened URLs.
Anonymous Shortening: The system may allow guest users to shorten URLs without creating an account, but these links will possess a default short expiration window (e.g., 30 days) and offer no dashboard tracking.
Analytics & Reporting
Click Tracking: The system must capture real-time usage data every time a short link is accessed.
Metric Dimensions: Collected data must include total click counts, timestamps, geolocations (country/city derived from IP addresses), referring domains (e.g., Twitter, LinkedIn), and user browser/device types.
2. Non-Functional Requirements
Non-functional requirements describe the operational constraints, performance targets, and quality attributes that the underlying infrastructure must guarantee.
High Availability and Fault Tolerance
Availability Target: The system must achieve a minimum of 99.99% availability (four nines), meaning no more than roughly 52 minutes of total downtime per calendar year.
Read vs. Write Asymmetry: The system will be highly read-heavy. The infrastructure must handle an expected read-to-write ratio of roughly 100:1 (many more people click links than create them).
Graceful Degradation: If the analytics logging cluster or database fails, the core link redirection engine must continue to operate uninterrupted using cached routes.
Scalability and Storage Constraints
Throughput Support: The system must smoothly handle an average of 10,000 redirection requests per second and 100 creation requests per second, with infrastructure capacity to handle peak bursts up to 5x average limits.
Storage Duration: The system must scale gracefully to persist billions of link records securely over a standard retention period of 10 years.
Memory Optimization: Active, heavily visited mapping records must be cached in-memory (e.g., via Redis) using an Least Recently Used (LRU) eviction policy to avoid hitting persistent databases for hot routing queries.
Latency and Performance Targets
Redirection Latency: The internal routing and redirection process must execute with sub-millisecond efficiency. The overall API response time for a short link redirection must not exceed 20 milliseconds (excluding external network transit delays).
Write Latency: Generating a new shortened URL must complete and return a payload within 150 milliseconds.
Security and Abuse Prevention
Rate Limiting: To prevent Distributed Denial of Service (DDoS) attacks and script abuse, strict rate limiting must be enforced per API key or IP address (e.g., maximum 100 link creations per minute per user).
Malicious URL Filtering: The system must integrate with automated security systems (such as Google Safe Browsing APIs) to scan incoming long URLs. Any link flagged for malware, phishing schemes, or spam must be blocked from shortening immediately.
Token Unpredictability: Generated short codes must be mathematically random or cryptographically masked to ensure users cannot guess sequential strings (e.g., discovering [https://pico.ly/aaaaaa2](https://pico.ly/aaaaaa2) simply by changing the last character of [https://pico.ly/aaaaaa1](https://pico.ly/aaaaaa1)).
